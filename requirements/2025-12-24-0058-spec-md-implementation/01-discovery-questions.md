## Q1: Botは「メンション付きメッセージ」のみをトリガーにしますか？

**Default if unknown:** Yes（SPECでメンション付きの挙動が明記されているため）
**Reasoning:** 現状のボットは機能が最小で、誤反応を避けるためにも明示的トリガーが自然です。

## Q2: Gemini CLIの結果は、SPECの例のように「tool_useの整形 + assistantメッセージ結合」で送る必要がありますか？

**Default if unknown:** Yes（SPECに具体的な整形例があるため）
**Reasoning:** 期待出力が具体化されているため、その形式に合わせるのが安全です。

## Q3: メッセージIDとセッションIDの対応は、プロセス再起動後も保持する必要がありますか？

**Default if unknown:** No（現状DB等がなく、最小実装はインメモリ保持が妥当なため）
**Reasoning:** 既存構成に永続化の仕組みが見当たらず、まずはインメモリ前提が最小変更です。

## Q4: Botの応答対象は、Botが参加している全てのギルド／チャンネルで有効にしますか？

**Default if unknown:** Yes（現在の実装はグローバルイベントを拾うため）
**Reasoning:** 既存のイベント購読は全体対象で、フィルタ実装が無い前提に沿います。

## Q5: Gemini CLIの同時実行数に上限（キューや排他）を設ける必要がありますか？

**Default if unknown:** No（小規模想定で、まずはシンプルに直列／無制限）
**Reasoning:** まだ利用規模や負荷要件がなく、制御は後から追加可能です。
